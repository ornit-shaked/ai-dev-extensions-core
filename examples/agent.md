# Adding a New Example - Agent Guide

**Purpose**: Guide for AI agents helping developers add workflow example outputs.

---

## When to Use This Guide

User requests: "Add an example for [workflow]" or "Create example output"

---

## Why Examples Matter

Examples demonstrate:
- Expected workflow output format
- Completeness levels (COMPLETE/PARTIAL/MISSING)
- Metadata tracking patterns
- Open issues documentation
- Agent-ready assessment

---

## Prerequisites

Before creating an example:
1. **Workflow must exist** (e.g., `architecture-intake-create`)
2. **Source data available** (real or synthetic)
3. **Domain identified** (architecture, security, etc.)
4. **Example name decided** (descriptive, e.g., `user-service-example`)

---

## Step-by-Step Process

### Step 1: Create Example Directory Structure

```bash
mkdir -p examples/{DOMAIN_NAME}/{EXAMPLE_NAME}/output
```

**Example**:
```bash
mkdir -p examples/architecture/payment-service-example/output
```

---

### Step 2: Reference Existing Example

**Study**: `examples/architecture/user-service-example/`

This example contains:
- 8 section files (1-identity.md through 8-observability.md)
- `_metadata.yaml` (completeness tracking)
- `_open-issues.md` (items needing review)
- `README.md` (explanation of example)

**Use this as your template!**

---

### Step 3: Generate Output Files

Based on the workflow (e.g., architecture-intake-create):

**Required files**:
```
examples/{DOMAIN}/{EXAMPLE}/output/
├── 1-identity.md         ← Section 1
├── 2-architecture.md     ← Section 2
├── 3-contracts.md        ← Section 3
├── 4-data.md             ← Section 4
├── 5-flows.md            ← Section 5
├── 6-errors.md           ← Section 6
├── 7-config.md           ← Section 7
├── 8-observability.md    ← Section 8
├── _metadata.yaml        ← Metadata summary
└── _open-issues.md       ← Open issues list
```

**Each section file must include metadata block at end**:
```yaml
---

## Metadata

```yaml
source: AUTO|SEMI|MANUAL
completeness: COMPLETE|PARTIAL|MISSING
needs_human: true|false
risk: LOW|MEDIUM|HIGH
last-updated: YYYY-MM-DD
notes: "Additional context"
```
```

### Step 4: Create _metadata.yaml

**File**: `examples/{DOMAIN}/{EXAMPLE}/output/_metadata.yaml`

**Template**:
```yaml
# Architecture Intake Metadata
# Generated by: {workflow-name} v{version}
# Service: {service-name}
# Generated at: {timestamp}

service:
  name: {service-name}
  display_name: {Service Display Name}
  type: microservice
  
collection:
  workflow_version: "1.0.0"
  generated_at: "{ISO-timestamp}"
  generated_by: {workflow-name}
  duration_seconds: {number}

sections:
  identity:
    source: SEMI
    completeness: PARTIAL
    needs_human: true
    risk: MEDIUM
    notes: "..."
    
  architecture:
    source: AUTO
    completeness: COMPLETE
    needs_human: false
    risk: LOW
    notes: "..."
  
  # ... (all 8 sections)

## Overall Statistics

total_sections: 8
complete_sections: 5
partial_sections: 3
missing_sections: 0
needs_human_review: 3

## Agent Consumption

agent_ready: PARTIAL|COMPLETE|NOT_READY
agent_ready_explanation: |
  Explanation of readiness level and limitations

confidence_level: LOW|MEDIUM|HIGH
confidence_notes: |
  Notes on confidence in the documentation

## File Manifest

files_generated:
  - "1-identity.md"
  - "2-architecture.md"
  # ... all files
  
total_files: 10
total_size_kb: {number}
```

## Validation Checklist

Before considering example complete:

- [ ] All required output files present
- [ ] Each section has metadata block
- [ ] `_metadata.yaml` created with all fields
- [ ] `_open-issues.md` documents all gaps
- [ ] Example `README.md` created
- [ ] Example follows existing patterns from domain
- [ ] File naming consistent (lowercase, hyphens)
- [ ] Content realistic and useful for learning

---

## Example Naming Convention

```
examples/
└── {domain}/
    └── {example-name}/
        ├── output/          # Generated files
        └── README.md        # Explanation
```

**Naming rules**:
- Lowercase with hyphens
- Descriptive (e.g., `user-service-example`, not `example1`)
- Include service type if relevant (e.g., `spring-boot-api-example`)

---

## Common Pitfalls

❌ **Incomplete metadata** - missing fields in _metadata.yaml  
❌ **No open issues** - every example should have some gaps  
❌ **Too perfect** - real examples have PARTIAL sections  
❌ **Missing README** - example needs explanation  
❌ **Inconsistent naming** - follow conventions
