# PRD: AI Dev Extensions Core

## Section 1: Introduction

**What**: A language-agnostic content package providing structured workflows, rules, and skills for AI-powered development tools (Windsurf, Cursor, VS Code). The package is distributed as a Git submodule and integrates into projects via symlinks, enabling AI agents to follow standardized processes such as architecture documentation, PRD creation, code review and etc.

**Why**: Development teams lack standardized, reusable workflows for AI agents working on microservices. Each project creates ad-hoc documentation approaches, leading to inconsistent quality, repeated work, and poor AI agent guidance. This package solves the problem by providing battle-tested workflows that AI agents can consume directly from their IDE.

**Goal**: Enable AI agents across multiple IDEs to produce consistent, high-quality documentation and follow best practices through standardized workflows,skills and rulles that are maintained centrally and distributed to all microservice projects.

---

## Section 2: Goals

1. **Standardize AI workflows** across all microservice projects (measure: 80% of projects use package within 6 months)
2. **Reduce documentation inconsistency** by providing templates and structured workflows (measure: 90% template adherence in generated docs)
3. **Enable multi-IDE support** so teams can choose their preferred AI tool (measure: Support 3+ IDEs by end of Phase 2)
4. **Centralize maintenance** of workflows to reduce per-project overhead (measure: 1 update propagates to all projects within 24 hours)

---

## Section 3: User Stories

1. **As a developer**, I want to integrate AI workflows into my project without writing custom workflows, so that I can focus on feature development rather than documentation infrastructure.

2. **As an AI agent**, I want to follow standardized workflows for architecture documentation so that I produce consistent, complete documentation across all projects.

3. **As a platform team**, I want to update workflows centrally and have changes propagate to all projects, so that improvements benefit all teams immediately.

4. **As a junior developer**, I want clear, step-by-step PRD documents generated by AI agents, so that I can implement features without ambiguity.

5. **As an architect**, I want AI-generated architecture documentation that follows our standards and includes all required sections, so that I can review and approve quickly.

---

## Section 4: Functional Requirements

1. The system must allow projects to integrate the package as a Git submodule
2. The system must provide automated setup scripts (PowerShell for Windows, Bash for Linux/Mac) that create symlinks based on IDE detection
3. The system must support multiple IDEs through configurable mappings in `config/ide-mapping.yaml`
4. The system must deploy workflows as individual file symlinks (flatten mode) to ensure IDE discovery
5. The system must deploy workflow assets (such as templates, scripts) to `.assets-{domain}/{workflow-name}/` directories accessible via relative paths
6. The system must organize content into domains (e.g., `_core`, `architecture`) with priority-based loading 
7. The system must allow selective domain enablement via `manifest.yaml` configuration
8. The system must provide metadata for each domain in `.domain-metadata.yaml` files
9. The system must support workflow versioning through frontmatter metadata
10. The system must include rules that constrain AI agent behavior (no-fabrication, minimal-changes)
11. The system must provide example outputs demonstrating correct workflow execution
12. The system must update `.gitignore` automatically to exclude generated/temporary files
13. The system must support dry-run mode for setup scripts to preview changes without modification
14. The system must maintain backwards compatibility for submodule updates (non-breaking changes)
15. The system must provide clear error messages when setup fails with actionable remediation steps
